## Website Operation Manual
Use this prompt when the task is to design, redesign, verify, or implement a webpage or website: landing pages, marketing pages, portfolios, brand sites, public content pages, campaign pages, static HTML website references, or multi-section public site templates. This manual is for websites, not webapps, dashboards, admin tools, data tools, editors, games, product task flows, or interaction-system design.

### Scope boundary
- A website is a public or content-led page/site whose primary job is communication, brand recall, conversion, publishing, or presentation. It may have simple navigation, links, forms, menus, image accordion, GSAP ScrollTriggered tabs, tabs, carousel-like presentation, and scroll narrative when they support the page story.
- A webapp is a task-running product UI with authenticated workflows, durable user state, dashboards, CRUD, data operations, editors, multi-step task flows, realtime collaboration, or complex interaction systems. Use Frontend for webapps and operational UI; use Website only for the webpage/site design layer.
- Do not turn a requested website into a webapp or interactive product shell. Do not invent dashboards, complex settings, fake app states, or task flows when the user asked for a webpage.
- If the user asks for both a public website and a webapp, separate the public website from the app experience and apply Website only to the public website pages.

### Website workflow
- For any greenfield website or webpage, establish the design in standalone HTML first or use an existing user-provided/reference HTML design as the source of truth. This still applies when the user asks for a specific technology stack, framework, package manager, UI library, or styling system: settle the standalone HTML design first, then implement that design in the requested stack.
- If the user has not clearly stated the webpage requirements, first make the standalone HTML webpage design, show or reference it for review, and ask for feedback or confirmation; do not start formal frontend service/app scaffolding, routing, or dev-server implementation until the website design direction is settled.
- Before redesigning an existing website, classify the mode as Greenfield, Preserve, or Overhaul. Preserve mode starts from the current brand, content, information architecture, routes, SEO, accessibility, and form semantics. Overhaul mode may change the visual language but still requires explicit protection for URLs, navigation labels, form field names, brand wordmark, legal copy, tracking semantics, and accessibility wins.
- For website implementation after the HTML reference is settled, preserve the design tokens, typography, grid, spacing, media crops, section rhythm, navigation behavior, responsive behavior, CTA priority, and content hierarchy in the requested stack.

### Website communication and content
- Every website section must have one clear communication job, one content focus, one primary visual element, and a reason to exist. It should advance the page narrative rather than behaving like a generic block stack.
- For marketing pages, landing pages, portfolios, and brand websites with no explicit style direction, use an Awwwards-level art-directed page style: distinctive section rhythm, real visual assets, deliberate asymmetry, generous whitespace, and a conversion path that still reads clearly. Do not apply this default to dashboards, admin tools, data tables, multi-step forms, code editors, or operational product UI.
- For marketing and landing pages, use AIDA as the default page narrative: Attention, Interest, Desire, Action. Default conversion path: hook -> proof/trust -> problem/solution -> product or service showcase -> benefits/use cases -> testimonial/evidence -> pricing or offer -> FAQ/objection handling -> final CTA. Do not add proof, pricing, metrics, or testimonials unless the content is real, source-grounded, or clearly framed as draft placeholder copy.
- For webpage landing-page heroes, make the H1 the brand/product/place/person name or a literal offer/category; put descriptive value props in supporting copy, not the headline.
- Do not add hero pill-tag clusters, generic hero stats, fake KPI walls, decorative system markers, section-number eyebrows, scroll cues, version footers, weather/time/location strips, filler testimonial names, repeated avatars, plain-text logo walls, or fake social links.
- Treat website copy as design material. Headings, labels, body copy, captions, helper text, CTA text, FAQ text, form text, and error text should explain, guide, persuade, or reduce friction without generic AI phrasing.

### Website visual system
- Use a disciplined layout with a 12-column desktop grid, maximum website content display width of 1440px, 4-column mobile grid, 64-96px desktop outer margins, 32-48px tablet margins, 20-24px mobile outer margins, 24-48px section gaps on dense pages, and 80-140px section gaps on open pages unless the reference requires otherwise.
- Full-bleed backgrounds may extend to the viewport, but readable/navigable content, media compositions, grids, and section layouts must stay within the 1440px maximum content width.
- For webpages, never place two adjacent page sections on merely adjacent or subtly different colors; adjacent sections must either use the same surface color or a deliberately high-contrast color shift.
- Navigation bars should hide when the user scrolls downward to increase readable viewport area. If a navigation bar is visible when the page is not at the very top, its width must be greater than or equal to the overall page content display width and use either a solid navigation surface or a frosted-glass treatment with visible fill, backdrop blur, edge separation, and verified text contrast over every section that can pass beneath it.
- Keep webpage typography deliberate and readable. Letter-based visible titles and headings must use letter-spacing strictly greater than `0.05em` and line-height strictly greater than `1`; treat these as lower bounds, not targets. If the selected font, language, or viewport makes those values harm readability, choose a different title composition or font instead of lowering them.
- Titles and headings must render within 3 lines at every supported viewport. If a title needs more room, rewrite it, split the section, or use intentional manual line breaks; do not allow accidental one-character-per-line wrapping.
- Define concrete webpage type variables before layout, such as `--type-display`, `--type-title`, `--type-section`, `--type-body`, and `--type-small`, and preserve at least 30% jumps between adjacent hierarchy levels.

### Website section structure
- Do not make a landing page unless the user asks for a landing page, marketing page, portfolio, brand site, campaign page, or public website. When the user asks for an app, game, tool, dashboard, or operational product, build the actual usable experience instead of a marketing page.
- If the user asks for a landing page and gives no section count, default to 6-9 sections. If the user asks for a full website template and gives no section count, default to 8 sections.
- Each webpage section must use an explicit fixed height or fixed viewport-height range for the designed breakpoint, not an auto-height block that merely expands with content. Define the section height before placing content; resolve overflow by shortening copy, changing the layout, or splitting the section rather than allowing clipping or uncontrolled growth.
- Hero sections must stay between 70vh and 92vh and use a strong primary object, product image, media composition, or scene plus iconic typography as the primary visual. The primary object or brand mark should occupy 20-45% of the hero area.
- Split text/media hero layouts are allowed when they best explain the product, service, portfolio, or conversion path. Make the split intentional with a clear grid, strong media crop, balanced whitespace, and a mobile collapse that preserves CTA visibility.
- When text overlays imagery, keep it directly on the image or scene and not in a card. Avoid card-on-one-side hero compositions unless the card is the actual product interface or tool surface, and never put hero text or the primary experience in a decorative card.
- Keep the primary CTA visible without scrolling, do not place pill tags under the hero, and do not place generic stats in the hero. Put proof and trust signals below the hero unless proof itself is the primary product artifact.
- On branded, product, venue, portfolio, or object-focused webpages, the brand/product/place/object must occupy at least 15% of the first viewport area or appear in text at 64px or larger on desktop and 32px or larger on mobile. Hero content must leave at least 80px of the next section visible on desktop and at least 48px visible on mobile.
- Every website section should use a distinct composition anchor and avoid repeating the same centered section, split text/image section, equal-card grid, or decorative block pattern.
- When many peer-level items need to be shown, choose an information pattern based on comparison and scanning needs: image accordion, tab-switching design, segmented panels, disclosure lists, comparison tables, curated indexes, timeline or stepper layouts, masonry or bento layouts, horizontal scroll-snap, carousel, marquee, dense lists, or split-feature layouts where one selected item drives a larger visual or explanation. Do not use the same design or interaction pattern in two adjacent sections.
- Do not use plain-text or text-only accordions to hide ordinary section copy, FAQs, feature lists, timelines, service lists, or generic content blocks. Use an accordion only when collapsed/expanded state is materially useful, and pair it with meaningful visual structure, media, comparison context, or progressive disclosure that improves scanning.
- When a bento layout is appropriate, build it as a gapless or near-gapless bento grid with intentional cell rhythm, shared edges, varied spans, and real content or imagery in every cell. Do not use scattered independent cards and call it bento.
- Treat the footer as a designed closing system, not a dump of small links. Choose a footer pattern that matches the page job: brand recall, CTA footer, sitemap footer, contact footer, support/legal footer, curated index, or compact utility footer. Repeat the page's primary CTA label rather than inventing a second CTA intent.

### Website references and media
- When asked to design a website and the user has not explicitly requested reference discovery or reference matching, default to this design brief: An Awwwards-level landing page with product/visual display and a clear "how it works" section. Requirements: AIDA page structure (Attention, Interest, Desire, Action); no pill tags under the hero; no generic stats in the hero; real GSAP ScrollTrigger motion for one pinned storytelling section.
- When the user explicitly asks for references, use image search to download visual references with keywords that include high-end design reference sites such as Awwwards and Mobbin. Split references between the task theme/style and website/page composition.
- When the user gives a specific brand, product, person, place, or object, prioritize official website design, official product images, official icons/logos, and official design systems before broader references.
- Websites must use visual assets. Search webpages and download usable media from page links, use known relevant images, or generate bitmap images for non-game visual deliverables. Primary images and media should reveal the actual product, place, object, state, or person; avoid dark, blurred, stock-like, or purely atmospheric media when the user needs to inspect the real thing.
- For generic temporary imagery in standalone webpage demos, use deterministic `https://picsum.photos/seed/...` images instead of Unsplash URLs. Do not use repeated media; each distinct person, product, place, or content block needs distinct imagery.
- For image-generation-driven website design, create one separate horizontal section image per section by default. Never compress many sections into one unreadable board. Each section image must communicate layout, hierarchy, spacing, typography scale, CTA priority, component styling, image treatment, and visual rhythm clearly enough that a developer can implement it directly.
- Across a final website composition, page sections should average at least one subject-relevant visual media resource each: real or generated image, product/media crop, icon set, chart, diagram, video, canvas/WebGL scene, or other relevant visual asset. Add generous vertical whitespace between major content sections and before the footer.

### Website validation
- For website and HTML webpage design, use the HTML-to-image script for all pages across exactly three viewport widths: `1440px`, `768px`, and `390px`; read and analyze the captured images to confirm styling, visibility, layout, text hierarchy, media placement, responsive behavior, and animation effects meet requirements and look polished.
- For greenfield frontend webpages, validation must cover both stages: capture and inspect the standalone HTML design reference before service/app implementation, then capture and inspect the implemented frontend app after it runs. The final check must name the HTML reference path, requested/implemented stack, final app URL or absolute HTML path, all covered pages/routes, the required `1440px`, `768px`, and `390px` viewport widths, interaction states, and build/type/lint/browser verification commands.
- Cover website interactions with Playwright screenshots at the same three viewport widths when they cannot be validated by the HTML-to-image script, including navigation states, menus, accordions, tabs, overlays, forms, hover/focus states, and motion states.
- Explicitly reject navigation that stays visible during downward scrolling without a deliberate exception, non-top visible navigation whose width is smaller than the overall page content display width, and any non-top visible navigation without a solid or frosted-glass surface. Navigation text must remain readable over every scrolled section.
- Verify that text and media do not overlap, clip, or get chopped. Text must keep at least 16px clearance in compact UI, 24px in normal content sections, 20px from mobile viewport edges, and 32px from desktop viewport edges.
- Inspect rendered title and heading styles and reject any letter-based visible title or heading whose computed `letter-spacing` is not strictly greater than `0.05` of its own font size or whose computed `line-height` is not strictly greater than `1` relative to its own font size.
- Reject desktop captures where readable/navigable website content display width exceeds 1440px; full-bleed backgrounds may extend wider, but the content shell, grid, media compositions, and navigation content must not.
- For public webpages, check semantic regions, meaningful alt text, title/description/OG meta, favicon, skip link when navigation is substantial, visible focus states, form validation and inline errors, loading/empty/error states, 404/back navigation where applicable, legal links and cookie consent where product context requires them, real destinations or disabled states for links, and no dead `#` buttons.
- During final verification, compare the output against downloaded references and confirm the subject direction, style direction, website format, and overall visual intent match those references.
