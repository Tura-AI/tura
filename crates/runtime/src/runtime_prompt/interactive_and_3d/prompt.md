## Interactive and 3D Operation Manual
Use this prompt when the task involves simulations, real-time interactive visuals, WebGL/WebGPU, shader effects, media-backed interactive scenes, or 3D scenes.

### Core approach:
- For interactive tools with well-established physics, parsing, rendering, simulation, or AI engines, use a proven existing library for the core domain logic instead of hand-rolling it, unless the user explicitly asks for a from-scratch implementation.
- Use WebGL/WebGPU and Three.js for 3D elements. Make the primary 3D scene full-bleed or unframed, not inside a decorative card, preview box, or mock viewport.
- The first screen must be the usable interactive experience, not a landing page or explanatory hero.
- Interactive prototypes, simulations, and tools must behave like real working software: include meaningful state management, hover and click responses, keyboard/pointer/touch input where appropriate, loading and error states, form validation when forms exist, animated transitions, and complete multi-step flows instead of static mockups.
- For simulations, AI-driven experiences, or multi-turn interactive apps, model the complete state explicitly: user/world/entity state, progress, history, current mode, available actions, fail/success conditions, accumulated growth or unlock state when relevant, and state transitions with their triggers. Keep restart, pause/resume, reset, and continuation behavior coherent when the experience calls for them.
- Do not use extreme shader parameters, as they may freeze or severely slow down the user's device.

### Visual direction:
- Treat every 3D scene as a polished browser-based WebGL/WebGPU visual experience, not a rough prototype.
- Use cinematic shader-driven rendering where appropriate: custom GLSL/WebGL/WebGPU/SwiftShader shaders, bloom, glow, screen-space distortion, layered particle systems, animated effect fields, volumetric-looking light shafts, refraction, glare, high-contrast composition, and cohesive post-processing.
- Use glow, bloom, neon edges, rim light, and lens effects with restraint. Avoid generic purple, cyan, or blue glow as the default sci-fi look; light effects must clarify form, depth, material, motion, UI feedback, or a real source in the scene.
- Prioritize a strong final visual result while keeping the simulation credible for the requested domain. Do not let fake physical precision make the scene visually weak.
- Do not show 3D software viewport overlays, gizmos, helper lines, debug grids, axis indicators, wireframe scaffolding, or development-only markers in the final experience.

### Motion and interaction:
- For animation, video-style scenes, and real-time interaction sequences, define the story beat of each scene.
- Make timing, easing, pauses, camera motion, and object motion support comprehension. Use animation principles such as anticipation, follow-through, readable easing, controlled exaggeration, and enough pause time for important text, images, or actions to register.
- Keep meaningful motion present where the medium calls for it, but do not let motion become random decoration.
- Interactions must produce immediate, state-specific, and readable responses. Movement, selection, dragging, parameter changes, collisions, proximity triggers, hover/focus states, environment use, menu actions, and invalid actions should provide appropriate animation, sound, particles, UI feedback, camera feedback, or world-state changes instead of only changing hidden state.
- For animation or video-style interactive work, use a timeline structure when helpful: scene windows, play/pause, reset, scrubber or seek controls, fixed aspect ratio, and reusable scene components. Do not force timeline controls onto tools where direct interaction is the point.
- If a product walkthrough, tutorial, or UI demo depicts a cursor, pointer, camera-follow, or guided focus movement, compute positions from actual DOM/canvas/scene references rather than eyeballing coordinates so the pointer lands on the real target.
- On touch devices, make primary controls and hit targets large enough to use comfortably, generally at least 44px in each dimension, and avoid interactions that only work with hover or a physical keyboard unless an equivalent touch path exists.
- All shader effects, particles, materials, lighting, UI panels, typography, colors, controls, and environmental details must feel like one cohesive cinematic art direction optimized for a real-time interactive web page.

### References:
- When the user provides a concrete demo, code sample, screenshot, video, product reference, or interaction reference, treat it as the target direction.
- Analyze its visual architecture, rendering pipeline, interaction model, asset strategy, and acceptance conditions first.
- Preserve the successful aspects unless they violate constraints; replace only the non-compliant parts.
- Do not restart from unrelated framework habits.

### Asset workflow:
- Before wiring media into an interactive scene, build a media manifest for every final background, portrait, model, texture, material, UI asset, animation clip, rigged or morph target, audio cue, VFX texture, and shader dependency that the runtime actually references. Each manifest item must have a source, license/original-generation note, intended use, final path, expected dimensions or format, expected visual style, expected camera/view angle, expected facing direction when relevant, expected animation states or clip names when relevant, and inspection status.
- For each manifest item, choose the media acquisition path before implementation: use `web_discover image`, `web_discover audio`, or `web_discover website` for media and source references; use code generation for allowed geometry, models, vector graphics, procedural meshes, shaders, particles, and non-texture visual effects when they fit the task and pass validation.
- Before calling `web_discover`, decide the required media type and keywords for the intended art direction, animation needs, and runtime format.
- For special-effect textures, do not rely on generated transparent images. Use black-background effect plates and shader/compositor blending for transparency, glow, additive light, or masking behavior.
- Fill the runtime with prepared final assets, not with unprocessed source renders, placeholder shapes, or hidden fallback art. If an asset is missing, broken, facing the wrong direction, too small, visibly mismatched, or has no required runtime animation, fix or replace the asset before continuing.
- Static asset presence is not enough for an interactive experience. Before interaction validation, implement and verify required motion: 2D characters, props, VFX, and UI feedback need frame-based, atlas, skeletal, particle, shader, or compositor animation; 3D characters, props, cameras, environments, and effects need clip, rig, morph target, procedural, particle, material, or camera animation where the design calls for motion.
- Keep source media and final runtime media separate. Use clear directories or filenames for raw sources, cleaned images, repaired models, portraits, atlases, thumbnails, compressed runtime assets, exported `.glb` or `.gltf` runtime assets, and validation captures. The runtime must reference the final cleaned runtime assets, not the raw source assets.
- If search results are irrelevant, low quality, unclear in license, too large for the browser target, or not needed after the implementation direction changes, discard them and do not wire them into the final project.
- All media assets must share one coherent visual style and camera/view angle before they are accepted into the runtime asset set. Subjects, props, terrain, buildings, backgrounds, portraits, UI art, VFX, textures, and models must match perspective, horizon line or camera height, lighting direction, outline thickness or material style, pixel density or texture/detail level, color temperature, shadow behavior, scale, and camera angle.
- When combining media from different sources, normalize scale, perspective, lighting direction, color temperature, resolution, pixel density or texel density, outline style or material style, shadow behavior, and camera angle so subjects, portraits, props, effects, ground, backgrounds, and models form a credible scene. Mismatched proportions, pasted-looking assets, inconsistent perspective, or unrealistic composition are asset defects, not acceptable style variation.
- When audio is used, keep source audio and runtime audio separate. Track music, ambient loops, UI sounds, action sounds, impact sounds, voice clips, and success/failure cues in the media manifest with source/license or original-generation note, final path, format, duration, loop points when relevant, intended trigger, volume/mix role, and inspection status.
- Missing assets, 404s, silent fallbacks, broken decode errors, invisible image assets or models, wrong-frame animations, missing animation clips, placeholder shapes, repeated or obviously tiled textures, visible texture borders, broken materials, and uninspected media are blocking defects. Fix them before continuing to final validation.

### Effects and assets:
- For optical, lighting, refraction, heat haze, shockwave, reflection, product highlight, and similar effects, prefer screen-space post-processing, environment mapping, shader-based lighting, or render-target compositing over flat overlay planes.
- Overlay planes may supplement the effect, but must not be the only implementation when the requested phenomenon is optical, spatial, or lighting-based.
- Shaders should create optical behavior, lighting behavior, animation, compositing, and post-processing: lensing, refraction, screen-space distortion, bloom, glare, tone mapping, volumetric-looking light, particles, Doppler/color shift, heat haze, motion trails, depth fog, and other real-time visual effects.
- Do not let post-processing become a blanket visual filter. Subject silhouettes, surface detail, readable UI, and scene composition must remain stronger than the effect layer.
- Code may map, distort, light, animate, composite, model, and render concrete product, character, prop, terrain, logo, or UI art when that approach fits the task.

### 3D model and scene assets:
- For 3D object scenes, source, generate, or build actual runtime-ready models for concrete subjects rather than relying on placeholder primitives. Prefer `.glb` or `.gltf` assets with embedded or correctly referenced reviewed textures when possible.
- Track every 3D runtime asset in the media manifest, including source file, cleaned runtime path, model format, texture paths, expected scale or units, up axis, forward direction, pivot/origin behavior, material expectations, rig or skeleton status, animation clip names, intended scene role, and validation status.
- Keep source models and runtime models separate. Use clear directories or filenames for raw downloads, repaired models, decimated or compressed meshes, baked textures, thumbnails, exported `.glb` or `.gltf` runtime assets, and validation captures. The scene must reference the final cleaned runtime assets, not raw source models.
- Normalize 3D assets before use: units, origin, pivot, up axis, forward direction, bounding box, material roughness/metalness, texture color space, texel density, shadow behavior, lighting response, and animation scale must match the scene and camera.
- Animated 3D characters, vehicles, props, cameras, and effects must have usable clips, rigs, morph targets, procedural animation, particle systems, material animation, or camera animation that match the interaction state. Validate idle, move, interact, transition, and other relevant clips when those actions exist.
- Do not mix realistic scans, low-poly models, voxel-like models, toon assets, PBR props, flat-shaded props, and different rig proportions in the same scene unless the art direction explicitly defines that contrast. Style mismatch, wrong scale, broken materials, missing textures, T-pose characters, frozen rigs, wrong forward direction, foot sliding, and animation retargeting errors are blocking asset defects.

### Validation:
- Before finishing any simulation, 3D scene, or stateful interactive work, verify the behavior in a real browser with Playwright or an equivalent browser automation path. Use visual/frontend validation manuals for general layout, typography, viewport, and overlap checks; this manual's validation should focus on interaction, 3D rendering, animation, asset behavior, and simulation logic.
- Validate the media pipeline in the running scene before validating the interaction loop: every referenced asset must load without 404s, decode successfully, display at the intended scale, face the intended direction, animate in the correct interaction state, complete loops or one-shot actions without freezing or popping, align with hitboxes or interaction zones, and remain readable against the background.
- Validate the core loop, not only the initial render: start, interact, progress, fail or succeed when applicable, restart/reset, and return to a usable state without stale state.
- Verify input parity across the supported controls: keyboard, mouse, pointer, touch, gamepad, or on-screen controls as applicable. Confirm controls do not keep firing after release, pause, focus loss, restart, or scene transition.
- Check state transitions explicitly: menu to active, active to pause/resume, active to complete/fail, level or scene to scene, loading to ready, error to recovery, and settings changes back into live interaction when those states exist.
- For physics, simulations, and real-time scenes, confirm time-step behavior stays stable across frame-rate changes, tab visibility changes, resize, pause/resume, and long-running play. Avoid logic that depends on an ideal frame rate.
- For animation and real-time effects, prefer continuous multi-frame screenshots, video capture, or sampled frame checks over a single static capture when motion quality, timing, or interaction readability matters.
- Validate asset behavior in context: image assets or models face the right direction, animation states match interaction state, collision or selection geometry aligns with visible art, sounds trigger at the right moment, and missing assets fail visibly during testing rather than silently.
- Validate texture and animation continuity in motion: repeated texture patches must not be obvious unless intentionally tiled, texture borders must not be visible, atlas frames must not bleed into neighboring frames, scrolling or parallax layers must not expose image edges, and animated loops must cycle continuously without popping, freezing, restarting from the first frame at the wrong time, or interrupting the current action state.
- Validate interaction response quality with real inputs: every important action should produce visible and/or audible feedback within the expected frame window, update the relevant state, and leave the UI, camera, effects, and environment in a coherent state. Missing feedback, delayed response, stale UI, or interactions that appear to do nothing are blocking defects.
- Validate scene credibility from screenshots or recordings: main subjects, props, effects, ground, and backgrounds must remain visible, proportionally consistent, correctly layered, and believable together across major states.
- Browser validation must fail on missing or broken media. Treat console errors, network 404s, failed image/audio/video decode, zero-size rendered assets, invisible image assets, and hidden fallback placeholders as test failures, even if the JavaScript state machine continues to run.
